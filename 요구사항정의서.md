## 1. 문서 목적

- 냠냠코치 프로젝트의 **필수/추가/심화 요구사항**을 명세서에 근거하여 확정한다.
- 각 요구사항에 대해 **REST 엔드포인트 ↔ UI 화면**을 매핑하고, 현재 구현 상태를 추적한다.
- 백엔드(Spring Boot) 본 구현 전, **필수 기능 중 누락된 항목**을 식별하여 우선순위를 정한다.

## 3. 기능적 요구사항

### 3.1 필수 기능 (Required)

| 번호 | 분류 | 요구사항명 | 요구사항 상세 | 우선순위 | 매핑(화면 / API) | 상태 |
|------|------|-----------|---------------|----------|------------------|------|
| F01 | 식단 | 식단 작성 | **사진 업로드 필수** → AI가 분석한 초안을 사용자가 수정 후 최종 등록. 항목은 **음식 DB(식약처)에서 선택**해 보정 | 필수 | MealLogView / `POST /api/meals/analyze` → `POST /api/meals` | 🟡 |
| F02 | 식단 | 식단 조회 | 식단 ID로 식단기록 내용을 상세 조회. 일자별 목록 조회 포함 | 필수 | MealHistoryView, MealDetailView / `GET /api/meals`, `GET /api/meals/{id}` | 🟡 |
| F03 | 식단 | 식단 수정 | 작성한 식단기록(항목·중량·메모 등)을 수정 | 필수 | MealDetailView / `PUT /api/meals/{id}` | 🔴 |
| F04 | 식단 | 식단 삭제 | 작성한 식단기록을 삭제(연관 항목·영양기록 함께 정리) | 필수 | MealDetailView / `DELETE /api/meals/{id}` | 🔴 |
| F05 | 식단 | 식단 분석 | 식단 기록을 **(공식 or AI)** 분석하여 영양 정보 표현 + **목표/질병에 맞춘 점수 산출** | 필수 | DashboardView, AiInsightsView / `GET /api/meals/{id}/analysis`, `GET /api/score` | 🟡 |
| F06 | 회원 | 회원 작성 | 회원가입으로 사용자 등록. 프로필(키, 몸무게, **질환** 등) 관리 | 필수 | LoginView, OnboardingView, ProfileEditView / `POST /api/auth/register` | 🔴 |
| F07 | 회원 | 회원 조회 | 회원 정보 조회 | 필수 | ProfileEditView, SettingsView / `GET /api/users/me` | 🟡 |
| F08 | 회원 | 회원 수정 | 회원 정보 수정 | 필수 | ProfileEditView / `PUT /api/users/me` | 🟡 |
| F09 | 회원 | 회원 삭제 | 회원 정보 삭제. 삭제 대신 **비활성화**로 상태 변경 가능 | 필수 | SettingsView / `DELETE /api/users/me`, `PATCH /api/users/me/deactivate` | 🔴 |
| F10 | 회원 | 로그인/로그아웃 | 로그인/로그아웃. JWT Access/Refresh 토큰 발급·재발급 | 필수 | LoginView / `POST /api/auth/login`, `POST /api/auth/logout` | 🟡 |

## 6. 구현 현황 요약

| 구분 | 완료 ✅ | 부분 🟡 | 미구현 🔴 | 향후 🔵 |
|------|---------|---------|-----------|---------|
| 필수(F01~F10) | 0 | 5 (F01·F02·F05·F07·F08·F10) | 4 (F03·F04·F06·F09) | - |
| 추가(F11~F15) | 0 | 0 | 5 | - |
| 심화 AI(F16~F17) | 0 | 2 | 0 | - |
| 향후 확장(F18~F21) | - | - | - | 4 |

**현황 진단**
- 필수 기능은 대부분 **프론트엔드 화면(UI mock)** 까지는 존재하나, **본 백엔드(Spring Boot + MySQL) 미연동**으로 영속성·인증이 동작하지 않는다.
- **식단 수정(F03)·삭제(F04)·회원 삭제(F09)** 는 화면·엔드포인트 모두 부재한 완전 미구현 상태이다.
- 회원가입(F06)은 **질환 프로필 필드 자체가 누락**되어 화면 보완이 추가로 필요하다.

### 의존성 핵심

```
회원가입+질환(3) → 식단작성(8) → 식약처DB(12) → 영양계산(13) → 점수산출(14)
                                                                   ↑
                                             질환 프로필(7) ────────┘
```

- **점수 산출(14)** 은 질환(7)·영양계산(13)을 모두 선행해야 하므로 이 둘을 먼저 완료한다.
- 현재 프로토타입은 심화(P4 AI 인식·AI 코치)가 앞서 있으나, 정작 필수인 **식단 수정·삭제(10·11)·점수 산출(14)** 이 비어 있다. **10·11·12·14를 최우선**으로 채운다.

---

> **결론**: dailyYam은 심화(AI 인식·AI 코치) 영역을 선제 구현한 강점이 있으나, 명세서가 요구하는 **필수 CRUD 완성(F03/F04/F09), 실제 인증(F06/F10), 질병 기반 점수 산출(F05), 공공 데이터 연동(식약처·국민건강통계)** 이 미충족 상태다. 백엔드(Spring Boot) 본 구현 시 §7의 우선순위 1번부터 순차 해소하고, 실제 코치(P5)는 그 이후 확장한다.
